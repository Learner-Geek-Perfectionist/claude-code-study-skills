# Study-Master Skill 深度分析增强设计

## 背景

当前 study-master skill 生成的文档在数据结构和函数分析方面存在以下不足：

1. **函数混淆**：大型开源项目中函数名高度相似（如 `ddsi_rmsg_addbias` vs `ddsi_rdata_addbias`），读者难以区分用途
2. **数据结构解析浅**：仅解释字段含义，未解释为什么需要该结构体、每个字段存在的合理性
3. **函数解析浅**：未说明函数的使用场景（何时调用、谁调用、什么目的），未做逐行精讲

## 设计决策

### 选择方案 B：重构模块章节为 9 节结构

**否决方案 A（在 7 节内增补）**：所有函数统一逐行精讲后 Section 3 会过长，塞在原结构中可读性差。

**否决方案 C（模板驱动）**：SKILL.md 膨胀严重，灵活性不足。

**选择方案 B 理由**：函数精讲独立成节，结构清晰；命名体系前置帮助建立全局视角。

### 全量深度分析

所有函数和数据结构统一按最高标准处理，不做核心/辅助分级。

| `ddsi_` | `rdata_` | `addbias` | 委托层：转调 rmsg_addbias，附加 debug 断言 |

末尾给出命名规律总结（1-3 条）。

### 2.2 易混淆函数对比表

列出名称相似的函数组，每组一张对比表：

| 对比维度 | `ddsi_rmsg_addbias` | `ddsi_rdata_addbias` |
|---------|---------------------|----------------------|
| 操作对象 | rmsg 的 refcount | 委托给 rmsg_addbias |

## Section 3：API Signatures（增强）

```c
static void ddsi_rmsg_addbias (struct ddsi_rmsg *rmsg);
  // 为 rmsg 的 refcount 增加一个 RDATA_BIAS 偏置
  // 场景：defrag 模块决定保留某个 rdata 时，由接收线程在 uncommitted 阶段调用
```

- 第一行：用途说明（保持原有要求）

## Section 4：数据结构深度解析（增强）

每个结构体的分析包含：

### 4.a 结构体存在的理由

在展示定义之前，先用引用块回答"为什么需要这个结构体"：

> **为什么需要 `struct ddsi_rbuf`？**
> 接收路径需要一块大的连续内存来存放网络数据。如果每条消息单独 malloc，
> 高吞吐场景下内存分配器会成为瓶颈。rbuf 提供预分配的大块内存，
> 用 bump allocator 实现 O(1) 分配。

### 4.b 结构定义

展示完整的 struct 定义源码（遵守真实代码忠实性规则）。

### 4.c 字段三层分析表

每个字段三列分析：

| 字段 | 设计动机（为什么需要） | 反事实（如果去掉会怎样） | 替代方案（还能怎么做） |
|------|----------------------|------------------------|----------------------|
| `n_live_rmsg_chunks` | 需要知道何时安全释放 rbuf | 无法判断 rbuf 是否仍被引用，可能 use-after-free | 可用链表追踪所有 chunk，但遍历开销 O(n) 远高于原子计数 O(1) |

### 4.d 生命周期状态图

使用 Mermaid stateDiagram-v2 展示结构体从创建到销毁的完整状态流转。

## Section 5：函数逐行精讲 [新增]

所有函数统一按以下格式做深度分析。

### 5.a 场景卡片

每个函数开头必须有：

> **函数：`ddsi_rmsg_rmbias_and_adjust`**
> - **调用时机**：reorder 阶段处理完一批 rdata 之后
> - **典型调用者**：`ddsi_fragchain_adjust_refcount` 遍历 fragment chain 时逐个调用
> - **前置条件**：rmsg 处于 uncommitted 状态，该 rdata 已经过 addbias
> - **目的**：一次原子操作完成"移除 BIAS 偏置 + 计入实际异步引用数"

### 5.b 逐行注释式精讲

展示函数完整源码，每一行代码后紧跟 `// →` 中文注释行：

```c
static void ddsi_rmsg_rmbias_and_adjust (struct ddsi_rmsg *rmsg, int adjust)
// → 函数签名：接收 rmsg 指针和 adjust（reorder 阶段累积的实际引用数）
{
// → 函数体开始
  uint32_t sub;
  // → 声明局部变量 sub，用于存储要减去的净值
  assert (adjust >= 0);
  // → 断言：adjust 不能为负，负值意味着 reorder 逻辑有 bug
  sub = RMSG_REFCOUNT_RDATA_BIAS - (uint32_t) adjust;
  // → 计算净减量：BIAS 偏置(2^20) 减去实际引用数，合并为一次减法
  if (ddsrt_atomic_sub32_nv (&rmsg->refcount, sub) == 0)
  // → 原子减法并返回新值；如果结果为 0，说明所有引用都已释放
    ddsi_rmsg_free (rmsg);
  // → refcount 归零，调用 free 释放 rmsg 占用的所有 chunk
}
// → 函数体结束；如果 refcount 非零，说明还有异步引用持有该消息，不做任何事
```

### 逐行注释规则

- 每一行源码（空行除外）后面必须有 `// →` 注释行
- 注释不仅要说"这行做了什么"，还要说"为什么这样做"或"如果不这样做会怎样"
- assert / trace / debug 宏也必须注释，解释它们在防御什么错误场景
- 函数最后一个 `}` 后附带一行总结性注释，说明"如果执行到这里意味着什么"
