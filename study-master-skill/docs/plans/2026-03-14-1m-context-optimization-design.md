# Study-Master 1M 上下文优化设计

**日期：** 2026-03-14
**状态：** 已批准
**目标：** 利用 Opus 4.6 的 1M 上下文 + 128K 输出能力，同时提升文档质量（特别是分析深度）和生成速度

## 背景

Study-master 当前架构因旧版上下文限制（~200K）做了以下妥协：

1. **SKILL.md 拆分为 4 文件**（主文件 160 行 + 3 个辅助文件），subagent 按需加载
2. **"指针而非内容"策略** — `.analysis-context.md` 只存文件路径和行号
3. **分段输出限制** — Write ≤50 行 + cat >> ≤80 行
4. **subagent 独立分析** — 各 subagent 各自 Read 源码做浅分析，缺乏全局视野

Opus 4.6 GA 后（2026-03-13），1M 上下文 + 128K 输出 + 无长上下文溢价，使得上述妥协不再必要。

## 方案选择

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A（选定）** | 深度分析前置 + 富上下文 Subagent | 质量速度双提升，架构改动适中 | .analysis-context.md 变大 |
| B | 全主 Dialog 生成 + 后置并行润色 | 质量最高 | 速度最慢，超大项目仍可能接近上限 |
| C | 两阶段 Subagent（分析+生成） | 速度最快 | 架构复杂，跨模块洞察可能遗漏 |

## 整体架构

```
新架构（1M 上下文）:

主 dialog          .analysis-context.md      subagent×N        主 dialog
(深度分析)    ───→  (富分析报告:             ───→ (纯生成,       ───→ (质量审查,
 1M 上下文          代码片段+解读)               快速输出)          交叉引用注入)
                                                                    ↓
                                                               高质量文档
```

对比旧架构：

```
旧架构（上下文受限）:

主 dialog          .analysis-context.md      subagent×N
(浅分析)     ───→  (指针表:                 ───→ (分析+生成,     ───→ 章节文件
                    路径+行号)                    各自 Read 源码)
```

## 详细设计

### 1. 主 Dialog 深度分析阶段（Stage 3-4 重构）

#### Stage 3：全源码深度分析

**旧流程：** Glob/Grep 找文件 → 提取函数签名 → 生成模块映射表
**新流程：**

1. **源码加载**：Read 项目所有核心源文件到 1M 上下文（可加载 ~30000 行代码）
2. **LSP 增强分析**（如可用）：
   - 对热点函数做 `callHierarchy` → 完整调用树
   - 对关键数据结构做 `findReferences` → 使用模式
3. **深度解读**（在 1M 上下文中完成）：
   - 每个模块的设计意图和职责边界
   - 关键算法的逻辑解读（不只是签名，而是设计动机）
   - 模块间的耦合关系和数据流
   - 关键设计决策和 tradeoff 分析

#### Stage 4：富分析上下文序列化

`.analysis-context.md` 从"指针表"升级为"富分析报告"：

```markdown
# 深度分析报告

## 项目全局架构
[整体设计理念、分层结构、核心数据流的深度解读]

## 模块 A：核心引擎
### 设计意图
[为什么存在这个模块，解决什么问题]
### 关键代码片段与解读
[核心函数的实际代码 + 逐段解释]
### 调用关系
[完整调用链，谁调用它、它调用谁、为什么]
### 设计决策
[为什么选择这种实现而非其他方案]

## 模块间关系图
[跨模块数据流、依赖关系、耦合分析]
```

### 2. Subagent 角色转变

**旧角色：** 分析 + 生成（4 次 Read 启动 + 自己分析源码）
**新角色：** 纯生成（2 次 Read 启动，专注文档写作）

#### Subagent Prompt 模板

```
你是 study-master 文档生成器。你的任务是将已完成的深度分析转化为教科书风格文档。

准备步骤：
1. Read {study_path}/.analysis-context.md — 获取完整的深度分析报告
2. Read {skill_path}/format-rules.md — 获取格式规范

你负责的章节：第 {N} 章 - {章节标题}
在分析报告中找到对应模块的完整分析内容，将其转化为教科书风格文档。

生成要求：
- 使用多层代码呈现（调用图 → 伪代码 → 真实代码 → 实现细节）
- 分析报告中已包含关键代码片段和解读，直接使用
- 如需更多源码细节，可以 Read 源文件（路径已在分析报告中提供）
- 单次 Write 输出完整章节
- 输出到：{study_path}/{NN}-{module}.md
```

#### 变化对比

| 维度 | 旧 | 新 |
|------|-----|-----|
| 启动 Read 次数 | 4 次 | 2 次 |
| 源码分析 | subagent 自己做 | 主 dialog 已完成 |
| 分析深度 | 浅（独立上下文） | 深（继承主 dialog 全局分析） |
| 输出方式 | 分段 Write ≤50 + cat ≤80 | 单次 Write 完整章节 |
| analysis-guide.md | 必读 | 不需要 |
| document-templates.md | 必读 | 可选 |

### 3. 质量增强 — 主 Dialog 后处理

subagent 全部完成后，主 dialog 利用 1M 上下文做内容级增强：

#### Step 1：全文加载与审查
- Read 所有生成的章节文件
- 审查内容质量：分析深度、逻辑清晰度、遗漏检查

#### Step 2：交叉引用注入
- 添加前后引用：
  - "这里使用的 `redisDb` 结构体将在[第 3 章](03-storage.md)中深入解析"
  - "该函数的调用者已在[第 1 章](01-command.md#processCommand)中讨论"

#### Step 3：深度补充
- 如 subagent 章节分析不足，主 dialog 用 Edit 工具直接补充
- 补充：缺失的设计决策分析、未解读的关键代码路径、跨模块交互的深层逻辑

#### Step 4：概览章节更新
- 根据所有章节实际内容更新 `00-overview.md`
- 确保架构图和学习路线图与各章节一致

#### 流程变化

```
旧 Finalization: 格式验证 → 修复 → 生成附录 → 完成
新 Finalization: 格式验证 → 内容审查 → 交叉引用注入 → 深度补充 → 概览更新 → 生成附录 → 完成
```

## 文件变更清单

| 文件 | 变更类型 | 变更内容 |
|------|----------|----------|
| **SKILL.md** | 重构 | Stage 3-4 深度分析流程；Stage 5-6 subagent prompt 重构；新增质量审查阶段；移除分段输出限制 |
| **analysis-guide.md** | 精简 | 移除 subagent 用的分析指导，保留 LSP 使用方法供主 dialog 参考 |
| **document-templates.md** | 更新 | `.analysis-context.md` 模板升级为"富分析报告"格式；简化 subagent 相关模板 |

| **hooks/validate_study_master.py** | 无变更 | 格式校验逻辑不变 |

## 速度提升预估

| 环节 | 旧耗时因素 | 新情况 | 提升 |
|------|-----------|--------|------|
| Subagent 启动 | 4 次 Read + 自己分析源码 | 2 次 Read，不需分析 | ~50% |
| 章节输出 | 多次 Write+cat（分段） | 单次 Write 完整章节 | ~60% |
| 主 dialog 分析 | 浅分析（快但质量低） | 深分析（稍慢但质量高） | -20% |
| 后处理 | 仅格式验证 | +内容审查+交叉引用 | -15% |
| **整体** | | | **净提升 ~20-30%，质量大幅提升** |
