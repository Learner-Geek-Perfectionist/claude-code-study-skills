# Study-Master Skill

## 概述

帮助程序员深度学习：
- 开源项目源码
- 协议栈实现（如 TCP/IP、HTTP）
- 特定编程语言或框架的内部机制

生成深度解析型学习文档，采用教科书风格。

**文档规模**：根据主题复杂度动态调整，通常 30-100 页。

**✨ 新特性**：自动检测并使用 LSP（Language Server Protocol）增强代码分析。

## 推荐目录结构

```
project-root/
├── src/                    # 源码目录
│   └── <module-name>/
├── docs/                   # 协议规范文档
│   └── <spec-name>/
└── study/                  # 生成的学习文档（自动创建）
    ├── <topic1>/
    └── <topic2>/
```

## 使用方式

```bash
/study-master <topic>                    # 自动查找源码
/study-master <topic> --source <path>    # 手动指定源码路径
```

## 教科书风格学习体验原则

生成的文档必须遵循渐进式知识展开：

1. **先整体后局部**：先展示模块在系统中的位置，再深入细节
2. **先接口后实现**：先讲清楚"做什么"，再讲"怎么做"
3. **先主流程后边界**：先追踪正常执行路径，再分析错误处理
4. **先概念后代码**：先用自然语言解释设计思想，再展示代码

## 工作流程

### 阶段 0：源码定位（自动）

**如果用户未指定 `--source` 参数，按优先级查找**：

1. `src/<topic>/` → `src/*<topic>*/` → `docs/<topic>/` → `docs/*<topic>*/` → `./<topic>/`
2. 不区分大小写，支持部分匹配
3. 找到多个候选时列出让用户选择
4. 文档统一输出到 `study/<topic>/`

---

### 阶段 1：LSP 环境检测（自动）

**C/C++ 项目自动检测 LSP 可用性**：

1. 查找 `compile_commands.json`（源码根目录、上级目录、build/ 目录）
2. 测试 LSP 响应：`LSP documentSymbol <file> line:1 char:1`
3. 记录结果：✓ LSP 可用 / ✗ LSP 不可用

**其他语言**：跳过 LSP 检测，使用传统工具。

---

### 阶段 2：主题识别与准备

- 解析 `<topic>` 参数，判断类型（项目/协议/语言内部机制）
- 确定源码位置
- 创建目标目录 `study/<topic>/`

---

### 阶段 3：项目结构分析

**根据 LSP 可用性选择分析方法**：

| 任务 | LSP 可用 | LSP 不可用 |
|------|---------|-----------|
| 获取符号列表 | `LSP documentSymbol` | Grep 搜索函数定义 |
| 识别核心 API | `LSP findReferences` 统计引用 | 基于文件位置推断 |
| 构建模块依赖 | 基于符号文件位置分组 | Glob/Grep 扫描 |

**LSP 深度分析（如果可用）**：

1. **热点函数识别**：按 `findReferences` 引用次数排序，识别最核心的函数
2. **模块耦合度分析**：统计跨文件引用，识别高耦合模块
3. **API 稳定性评估**：公共 API 的引用广度（被多少文件调用）
4. **调用深度分析**：使用 `prepareCallHierarchy` + `incomingCalls` + `outgoingCalls` 构建完整调用树

**输出**：项目架构概览、核心模块列表、热点函数排名

---

### 阶段 4：生成学习大纲

- 基于分析结果，确定章节划分
- 识别核心模块和学习路径
- 生成文档目录结构

---

### 阶段 5：生成快速导览

编写 `00-overview.md`，包含：
1. 项目简介（1-2页）
2. 核心概念速览（2-3页）
3. **典型场景剖析**（5-10页）- 必须包含完整执行路径追踪：
   - **调用栈可视化**：从入口函数到底层实现的完整调用链（Mermaid graph）
   - **数据流追踪**：关键数据结构在各函数间的传递和变化
   - **时序图**：多模块协作时的交互顺序（Mermaid sequenceDiagram）
   - **状态转换**：对象/系统状态的变化过程（如适用）
4. 架构全景图（1页）- Mermaid 图表
5. 学习路线图（1页）

---

### 阶段 6-N：逐模块深度解析

**⚠️ 重要：必须生成足够的章节以充分解释主题**

- 在阶段 4 生成大纲时，明确列出**所有**需要生成的章节
- **必须逐个生成所有章节**，不能中途停止
- 每个章节根据内容复杂度调整长度（通常 10-25 页）
- 完成标准：核心概念、关键函数、典型场景都已覆盖

**对每个核心函数，根据 LSP 可用性选择分析方法**：

#### LSP 可用时

1. **获取函数签名**：`LSP hover <file> line:<line> char:<col>`
2. **查找调用关系**：`LSP findReferences <file> line:<line> char:<col>`
3. **构建完整调用树**：
   - `LSP prepareCallHierarchy <file> line:<line> char:<col>` - 准备调用层次
   - `LSP incomingCalls <file> line:<line> char:<col>` - 谁调用了这个函数
   - `LSP outgoingCalls <file> line:<line> char:<col>` - 这个函数调用了谁
4. **生成执行路径追踪**：
   - 使用 Mermaid sequenceDiagram 展示函数调用顺序
   - 标注每个关键点的参数值、返回值、状态变化
   - 用注释区分：正常路径、错误处理、边界条件
5. **生成文档**，标注数据来源：
   ```markdown
   **函数签名**（来自 LSP hover）：
   **调用关系**（共 X 次引用，来自 LSP findReferences）：
   **调用树深度**：N 层（来自 LSP callHierarchy）
   ```

#### LSP 不可用时

1. 使用 Grep 搜索函数定义和调用
2. 从头文件提取函数签名
3. 手动推断调用关系
4. **生成文档**，标注数据来源：
   ```markdown
   **函数签名**（从头文件提取）：
   **调用关系**（基于 Grep 搜索）：
   ```

---

### 最后阶段：整合与收尾

**⚠️ 只有满足以下条件才能进入此阶段**：
- ✅ 已生成所有计划的章节
- ✅ 核心概念、关键函数、典型场景都已覆盖
- ✅ 每个章节都已通过 Read 检查，无乱码

**执行步骤**：
- 生成参考资料索引 `appendix-references.md`
- 创建 `.study-meta.json` 元数据（见下方格式）
- 更新 README.md，标记所有章节为"已完成"
- 输出最终报告：
  ```
  ✅ 学习文档生成完成！
  📍 位置: study/<topic>/
  📊 总页数: X 页
  📚 章节数: Y 个
  🔧 LSP 增强: 已使用/未使用
  ```

## 文档结构

### 快速导览章节（00-overview.md）

1. 项目/协议简介（1-2页）
2. 核心概念速览（2-3页）
3. 典型场景剖析（5-10页）
4. 架构全景图（1页）
5. 学习路线图（1页）

### 模块深度解析章节（01-module-xxx.md）

每个模块文档包含：

1. **模块概述**（1-2页）
2. **多层次代码展示**（10-15页）
   - 调用关系图（Mermaid）
   - 伪代码
   - 真实代码片段
   - 实现细节
3. **数据结构深度解析**（3-5页）
4. **关键算法剖析**（5-8页）
5. **设计决策分析**（2-3页）
6. **学习检查点**（1页）
   - **📝 本章小结**：3-5 个要点总结
   - **🤔 思考题**：2-3 个引导性问题，帮助读者检验理解
     - 例如："如果去掉 X 函数的 Y 参数检查，会发生什么？"
     - 例如："为什么这里使用 Z 数据结构而不是 W？"

## 文档生成规范

**严格遵守** `~/.claude/skills/_shared/format-rules.md` 中的所有规范：

- 源码位置：`> 📍 源码：[文件名:起始行-结束行](相对路径#L起始行)`
- 强制使用 Mermaid，禁止 ASCII art
- 节点文本 ≤ 25 字符，同层节点 ≤ 6 个，总节点 ≤ 20 个
- 表格上方添加图注：`> **图 N** 标题——描述`
- 纯文本标识符用反引号，超链接标识符不加反引号
- 代码块必须指定语言标识（c/bash/mermaid/text）
- 数学表达式使用 LaTeX 格式（`$...$`），禁止 Unicode 数学符号

### 可视化类型选择指南

根据内容选择合适的 Mermaid 图表类型：

| 内容类型 | 使用图表 | 示例场景 |
|---------|---------|---------|
| 模块依赖、调用关系 | `graph TD/LR` | 函数调用链、模块架构 |
| 时序交互 | `sequenceDiagram` | 多模块协作、执行路径追踪 |
| 状态转换 | `stateDiagram-v2` | 对象生命周期、协议状态机 |
| 数据结构 | `classDiagram` | 结构体成员、类层次 |

## 质量控制

### 内容质量
- 每个模块章节根据复杂度调整长度（通常 10-25 页）
- 生成后逐文件用 Read 检查乱码
- 发现 U+FFFD 或格式问题立即修正
- 确保所有源码引用都有正确的超链接
- 验证 LSP 数据准确性（如果使用了 LSP）

### 分段输出规则（重要）

**⚠️ 必须遵守分段输出策略，避免一次性输出过多内容**：

1. **初始文件创建**：使用 Write 工具，内容 ≤ 50 行
2. **后续内容追加**：使用 Bash `cat >>` 命令，每次 ≤ 80 行
3. **大章节生成**：分 3-5 次追加完成，而不是一次性写入

**示例**：
```bash
# 正确做法
Write 文件头部（~50 行）
→ Bash cat >> 追加第一部分（~80 行）
→ Bash cat >> 追加第二部分（~80 行）
→ Bash cat >> 追加第三部分（~80 行）

# 错误做法
Write 整个文件（200+ 行）  # ❌ 违反规则
```

**原因**：
- 提高用户体验（可以看到进度）
- 避免工具超时
- 便于 hook 验证每个输出段
